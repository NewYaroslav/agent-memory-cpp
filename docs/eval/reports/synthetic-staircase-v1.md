## Interpretation

On this deterministic lexical generator, `bm25_exact` is the better
dependency-free control baseline at every measured size. It has higher
Recall@10, nDCG@10, and MRR than `bow_vector`, while also keeping lower p95
latency.

The scaling shape is the useful signal:

- `bow_vector` p95 latency grows from 4.486 ms at 1k documents to 70.603 ms at
  5k documents;
- `bm25_exact` p95 latency grows from 1.076 ms at 1k documents to 8.361 ms at
  5k documents;
- `bow_vector` quality degrades as the corpus adds more lexically overlapping
  distractors. The current raw term-frequency cosine representation lacks
  BM25-style IDF weighting and document-length normalization. The growing dense
  vocabulary additionally explains its unfavorable memory and latency scaling,
  but not by itself the quality decline.

This supports treating `bm25_exact` as the primary no-dependency lexical
baseline for the next retrieval experiments. `bow_vector` remains useful as a
small-fixture smoke/control path, but it should not be used as the scalability
baseline for roadmap-label PR #29 candidate-filter work.

## Notes

- This is a synthetic lexical sanity sweep, not a production-quality
  benchmark or acceptance threshold.
- Timing values come from one local run and are directional, not statistically
  stable benchmark results.
- `bow_vector` uses dense bag-of-words vectors with brute-force exact
  top-K search, so growth in corpus size also increases vector
  dimensionality for this generator.
- Per-baseline peak RSS remains `0` for in-process synthetic sweeps;
  memory attribution requires subprocess isolation or another
  dedicated measurement path.
